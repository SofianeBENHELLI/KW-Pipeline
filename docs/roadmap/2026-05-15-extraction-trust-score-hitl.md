## Purpose

This document describes how to compute a confidence and trust score for semantic extraction from PDF, Word, PowerPoint, and template documents, including pages or slides containing images, diagrams, charts, tables, screenshots, and other visual elements.

The goal is to detect when the extraction is reliable enough to be automatically accepted, and when it should trigger a Human-in-the-Loop review workflow.

This design complements the semantic extraction pipelines already defined for:

- PDF documents
- Word documents
- PowerPoint documents
- PowerPoint templates: `.pot` / `.potx`
- Mixed documents containing text, tables, visuals, screenshots, and diagrams

---

# 1. Core concept

## Extraction Trust Score — ETS

**Extraction Trust Score**, or **ETS**, is a composite score estimating how reliable the extracted document knowledge is.

It is not just an LLM confidence score.

It combines several signals:

- Text extraction quality
- Layout preservation quality
- Visual extraction quality
- Semantic consistency
- Source traceability
- Model agreement
- Risk profile of the extracted object

The score is computed at multiple levels:

```text
Document
 ├── Page / Slide
 │    ├── Text block
 │    ├── Table
 │    ├── Visual zone
 │    ├── Chart
 │    ├── Diagram
 │    └── Extracted key information
 └── Global extraction result
```

The important design principle is:

> Do not only score the full document. Score every meaningful extracted object.

A document can have a high global confidence but still contain a risky diagram, a poorly extracted table, or an unsupported business claim.

---

# 2. Why a confidence score is required

Semantic extraction pipelines can fail silently.

Typical failure cases:

- Scanned PDF pages are partially OCRed.
- Two-column layouts are read in the wrong order.
- A PowerPoint architecture diagram is summarized too vaguely.
- A chart is interpreted without the axis or legend being properly read.
- A table embedded as an image loses its row and column structure.
- A diagram arrow is interpreted in the wrong direction.
- A key claim is generated by the LLM but cannot be traced back to the source.
- A visual contains important information that is ignored because there is little text.
- A screenshot contains another document or application UI that needs specific handling.

The confidence mechanism must therefore answer:

```text
Can this extracted information be trusted enough to be reused automatically?
```

If the answer is no, the system should create a HITL validation task.

# 3. Score hierarchy

ETS must be computed at five levels.

## 3.1 Document-level score

Measures the overall extraction reliability for the full document.

Example:

```json
{
  "document_id": "doc_001",
  "object_type": "document",
  "extraction_trust_score": 0.82,
  "risk_level": "medium",
  "hitl_required": true,
  "reason": [
    "Two slides contain complex diagrams with low relationship confidence",
    "One chart contains decision-grade numbers requiring human validation"
  ]
}
```

## 3.2 Page / slide-level score

Measures the extraction reliability for each page or slide.

Example:

```json
{
  "document_id": "doc_001",
  "page": 8,
  "slide": null,
  "object_type": "page",
  "extraction_trust_score": 0.61,
  "risk_level": "high",
  "hitl_required": true,
  "reason": [
    "Scanned page detected",
    "Low OCR confidence",
    "Dense table as image"
  ]
}
```

## 3.3 Chunk-level score

Measures the reliability of a semantic chunk.

Example:

```json
{
  "chunk_id": "chunk_017",
  "object_type": "chunk",
  "extraction_trust_score": 0.76,
  "risk_level": "medium",
  "hitl_required": false,
  "reason": [
    "Chunk is traceable and semantically consistent",
    "No high-risk visual dependency"
  ]
}
```

## 3.4 Visual-zone score

Measures the reliability of information extracted from a visual area.

Example:

```json
{
  "visual_id": "visual_008",
  "object_type": "visual_zone",
  "visual_type": "architecture_diagram",
  "extraction_trust_score": 0.46,
  "risk_level": "high",
  "hitl_required": true,
  "reason": [
    "Dense architecture diagram",
    "Low OCR confidence on labels",
    "Many directional arrows detected",
    "Relationship extraction uncertain"
  ]
}
```

## 3.5 Key-information score

Measures the reliability of each extracted claim, decision, requirement, KPI, risk, action, or definition.

Example:

```json
{
  "info_id": "info_024",
  "object_type": "key_information",
  "information_type": "requirement",
  "content": "The platform must support external model execution through API calls.",
  "extraction_trust_score": 0.88,
  "risk_level": "low",
  "hitl_required": false,
  "source_chunk": "chunk_017",
  "source_location": {
    "document_id": "doc_001",
    "slide": 7,
    "bbox": [120, 220, 850, 610]
  }
}
```

# 4. Global scoring formula

## 4.1 Default formula

The default ETS formula is:

```text
ETS =
  0.25 × Text Extraction Quality
+ 0.20 × Layout Preservation Quality
+ 0.20 × Visual Extraction Quality
+ 0.15 × Semantic Consistency
+ 0.10 × Source Traceability
+ 0.10 × Model Agreement
```

Each component is scored from `0` to `1`.

Example:

```json
{
  "object_id": "page_008",
  "object_type": "page",
  "confidence": {
    "overall": 0.64,
    "text_quality": 0.82,
    "layout_quality": 0.66,
    "visual_quality": 0.42,
    "semantic_consistency": 0.71,
    "traceability": 0.94,
    "model_agreement": 0.58
  },
  "risk": {
    "level": "medium",
    "hitl_required": true,
    "reasons": [
      "Visual extraction confidence below threshold",
      "Complex diagram with uncertain relationships"
    ]
  }
}
```

## 4.2 Adapted weighting by content type

The scoring weights should change depending on the object type.

### Native text chunk

```text
ETS_text_chunk =
  0.35 × Text Extraction Quality
+ 0.20 × Layout Preservation Quality
+ 0.20 × Semantic Consistency
+ 0.15 × Source Traceability
+ 0.10 × Model Agreement
```

Visual extraction is not relevant unless the chunk depends on a visual.

### Visual zone

```text
ETS_visual_zone =
  0.30 × Visual Extraction Quality
+ 0.20 × OCR Quality
+ 0.20 × Structure Extraction Quality
+ 0.15 × Semantic Consistency
+ 0.10 × Source Traceability
+ 0.05 × Model Agreement
```

### Table image

```text
ETS_table_image =
  0.25 × OCR Quality
+ 0.25 × Table Structure Quality
+ 0.20 × Cell Alignment Confidence
+ 0.15 × Semantic Consistency
+ 0.10 × Source Traceability
+ 0.05 × Model Agreement
```

### Chart

```text
ETS_chart =
  0.20 × Visual Extraction Quality
+ 0.20 × Axis / Legend Detection
+ 0.20 × Data Extraction Confidence
+ 0.15 × Business Interpretation Confidence
+ 0.15 × Source Traceability
+ 0.10 × Model Agreement
```

### Architecture diagram

```text
ETS_diagram =
  0.20 × Visual Extraction Quality
+ 0.20 × OCR Label Confidence
+ 0.25 × Relationship Extraction Confidence
+ 0.15 × Hierarchy / Layer Detection
+ 0.10 × Semantic Consistency
+ 0.10 × Source Traceability
```

# 5. Scoring dimensions

## 5.1 Text Extraction Quality

Text Extraction Quality measures how reliable the raw text extraction is.

### Signals

- OCR confidence
- Text density
- Ratio of readable characters
- Ratio of broken words
- Encoding quality
- Language detection consistency
- Noise level from headers, footers, watermarks, page numbers
- Missing paragraphs
- Repeated text blocks
- Unexpected empty pages

### Example formula

```text
Text Quality =
  0.40 × OCR Confidence
+ 0.20 × Clean Text Ratio
+ 0.20 × Language Consistency
+ 0.20 × Low Noise Ratio
```

### Example output

```json
{
  "text_quality": {
    "score": 0.87,
    "ocr_confidence": 0.91,
    "clean_text_ratio": 0.88,
    "language_consistency": 0.95,
    "noise_ratio": 0.12,
    "issues": []
  }
}
```

### Risk indicators

Trigger a warning if:

```text
OCR confidence < 0.75
Clean text ratio < 0.80
Language confidence < 0.80
Unreadable character ratio > 0.10
Text density unexpectedly low
```

---

## 5.2 Layout Preservation Quality

Layout Preservation Quality measures whether the document structure was correctly understood.

### Signals

- Heading hierarchy detected
- Slide titles detected
- Paragraph grouping confidence
- Reading order confidence
- Table boundaries preserved
- Figure boundaries detected
- Bounding boxes available
- Sections correctly separated
- Multi-column layout handled properly
- No major overlap between extracted blocks

### Example output

```json
{
  "layout_quality": {
    "score": 0.62,
    "reading_order_confidence": 0.55,
    "heading_detection_confidence": 0.72,
    "table_detection_confidence": 0.61,
    "bounding_box_coverage": 0.93,
    "issues": [
      "Two-column layout detected with uncertain reading order"
    ]
  }
}
```

### Risk indicators

Trigger a warning if:

```text
Reading order confidence < 0.70
Heading detection confidence < 0.60
Table detection confidence < 0.75
Bounding box coverage < 0.90
Multi-column layout is detected
Overlapping text blocks are detected
```

---

## 5.3 Visual Extraction Quality

Visual Extraction Quality measures how reliably the system extracted meaning from images, diagrams, screenshots, charts, tables, and illustrations.

### Visual types

The system should classify visual zones into one of the following types:

```text
image_with_text
screenshot
diagram
architecture_diagram
flowchart
chart
table_image
photo
illustration
unknown_visual
```

### Risk factors

Visual extraction is risky when the visual contains:

- Many arrows
- Many small labels
- Dense diagrams
- Complex hierarchy
- Low image resolution
- Low contrast
- Small text
- Screenshot of another document
- Table as image
- Chart without source data
- Overlapping shapes
- Handwritten annotations
- Multiple languages
- Rotated text
- Cropped image
- Watermark overlay

### Visual scoring formula

```text
Visual Extraction Quality =
  0.25 × OCR Confidence On Visual Text
+ 0.20 × Image Resolution Quality
+ 0.20 × Visual Type Confidence
+ 0.20 × Structure Extraction Confidence
+ 0.15 × Multimodal Summary Consistency
```

### Example output

```json
{
  "visual_id": "visual_008",
  "visual_type": "architecture_diagram",
  "visual_quality": {
    "score": 0.46,
    "ocr_confidence": 0.51,
    "image_resolution_quality": 0.72,
    "visual_type_confidence": 0.89,
    "structure_extraction_confidence": 0.38,
    "summary_consistency": 0.44,
    "issues": [
      "Many arrows detected",
      "Low confidence on diagram relationships",
      "Small OCR labels"
    ]
  }
}
```

---

## 5.4 Semantic Consistency

Semantic Consistency measures whether the extracted information is coherent with the rest of the document.

### Signals

- Topic appears in several supporting chunks
- Extracted summary is supported by the source text
- Extracted claims have evidence
- No contradiction between chunks
- Visual summary aligns with surrounding text
- Definitions are stable across the document
- Extracted entities match document vocabulary
- The slide/page title is aligned with the extracted visual content

### Example

If a slide title says:

```text
AI Openness Architecture
```

But the visual extractor summarizes the slide as:

```text
Financial reporting workflow
```

Then semantic consistency should be low.

### Example output

```json
{
  "semantic_consistency": {
    "score": 0.57,
    "topic_alignment": 0.62,
    "summary_source_support": 0.54,
    "visual_text_alignment": 0.45,
    "contradiction_detected": false,
    "issues": [
      "Visual summary weakly aligned with slide title"
    ]
  }
}
```

---

## 5.5 Source Traceability

Source Traceability measures whether each extracted piece of information can be traced back to the original document location.

### Required source fields

Every extracted object should have:

```json
{
  "source_location": {
    "document_id": "doc_001",
    "page": 5,
    "slide": null,
    "section": "Battery / Aging model",
    "bbox": [120, 300, 850, 620]
  }
}
```

For PowerPoint:

```json
{
  "source_location": {
    "document_id": "deck_001",
    "slide": 8,
    "shape_id": "shape_17",
    "bbox": [90, 120, 1020, 710]
  }
}
```

### Traceability scoring

```text
Traceability =
  1.0 if exact document + page/slide + bbox/shape + source chunk exists
  0.8 if document + page/slide + chunk exists
  0.6 if document + section exists
  0.3 if document only exists
  0.0 if no source exists
```

### Hard rule

Any extracted claim, requirement, KPI, decision, or risk with no source location must be flagged for HITL.

---

## 5.6 Model Agreement

Model Agreement measures whether multiple extraction methods agree.

This is especially useful for risky visual content.

### Examples of model agreement sources

- Document parser output
- OCR output
- Layout detector output
- Multimodal model summary
- LLM consistency checker
- Graph relationship extractor
- Rule-based validators

### Example

```json
{
  "claim": "The solution uses MCP for agent orchestration.",
  "model_agreement": {
    "score": 0.52,
    "supporting_extractors": ["multimodal_model"],
    "conflicting_extractors": ["ocr_engine", "text_parser"],
    "reason": "Only one extractor detected MCP; OCR detected unclear acronym."
  }
}
```

### Agreement scoring idea

```text
Model Agreement = number of extractors supporting the same result / total relevant extractors
```

Weighted model agreement can be used when some extractors are more reliable than others for a given content type.

# 6. HITL trigger logic

Use both:

1. Score thresholds
2. Hard rules

Do not rely only on the global score.

---

## 6.1 Default score thresholds

```text
ETS >= 0.85
→ Auto-approved

0.70 <= ETS < 0.85
→ Accepted with warning or sampled review

0.50 <= ETS < 0.70
→ HITL required for risky chunks, visuals, tables, claims, or relationships

ETS < 0.50
→ Full human review required for the object
```

---

## 6.2 Thresholds by document criticality

Different document types require different thresholds.

```text
Generic marketing document: 0.70
Internal knowledge document: 0.75
Customer-facing technical document: 0.80
Strategic business document: 0.80
Engineering specification: 0.85
Legal / contractual document: 0.90
Safety-critical document: 0.95
```

---

## 6.3 Hard HITL triggers

Even if the global score is high, trigger HITL when:

```text
Visual zone confidence < 0.60
Table image confidence < 0.75
Chart contains decision-grade numbers
Key requirement confidence < 0.80
Business claim confidence < 0.75
KPI confidence < 0.85
Decision confidence < 0.85
Claim has no source location
Diagram relationship confidence < 0.70
Contradiction detected between extracted chunks
OCR confidence < 0.70 on important pages
Scanned page detected in strategic document
Handwritten notes detected
Unknown visual type detected with relevant size
Page contains visual zones representing more than 50% of the content
```

---

## 6.4 Risk-level computation

Risk level should combine score and content criticality.

```text
Low risk:
- ETS >= 0.85
- No hard trigger
- Full traceability

Medium risk:
- 0.70 <= ETS < 0.85
- Or minor hard trigger
- Or weak model agreement

High risk:
- ETS < 0.70
- Or critical hard trigger
- Or missing source for important information

Critical risk:
- ETS < 0.50
- Or legal / safety / contractual content with low confidence
- Or contradiction in key extracted information
```

# 7. HITL workflow

## 7.1 Workflow overview

```text
Extraction pipeline
      ↓
Compute confidence scores
      ↓
Detect risky zones / chunks / claims
      ↓
Create HITL validation task
      ↓
Human reviewer sees:
  - Original page / slide
  - Highlighted source region
  - Extracted text / topic / claim / relation
  - Confidence score
  - Reason for low confidence
      ↓
Reviewer approves / edits / rejects
      ↓
Validated result is stored
      ↓
Feedback improves future extraction
```

---

## 7.2 HITL task types

Recommended task types:

```text
text_validation
ocr_validation
table_validation
chart_validation
diagram_validation
relationship_validation
claim_validation
requirement_validation
kpi_validation
decision_validation
risk_validation
topic_validation
summary_validation
```


## 7.4 Reviewer UI requirements

The reviewer interface should show:

- Original page or slide rendering
- Highlighted bounding box
- Extracted text
- Extracted summary
- Extracted key information
- Extracted entities
- Extracted relationships
- Confidence score breakdown
- Reasons for review
- Suggested reviewer action
- Previous similar corrections if available

The reviewer should be able to:

```text
Approve extraction
Edit extracted text
Edit topic label
Edit summary
Edit claim
Edit relationship
Edit table structure
Edit chart interpretation
Reject extraction
Mark visual as decorative
Mark information as non-relevant
Escalate to domain expert
```

# 8. Visual extraction and scoring

Visual extraction needs special handling because OCR alone is not enough.

## 8.1 Visual extraction pipeline

```text
Render page / slide as image
        ↓
Detect visual regions
        ↓
Classify visual type
        ↓
Apply specialized extractor
        ↓
Generate semantic visual chunk
        ↓
Score extraction confidence
        ↓
Trigger HITL if needed
        ↓
Attach visual chunk to document topics
```

## 8.3 Visual-specific HITL triggers

### Diagram

Trigger HITL if:

```text
Relationship confidence < 0.70
More than 10 nodes detected
More than 8 arrows detected
OCR confidence on labels < 0.75
Diagram type is unknown
Diagram summary conflicts with slide title
```

### Chart

Trigger HITL if:

```text
Chart contains numeric values used in extracted claims
Axis labels are not detected
Legend is not detected
Data extraction confidence < 0.80
Business interpretation confidence < 0.75
Chart is used to extract KPI, benchmark, revenue, cost, performance, adoption, or compliance information
```

### Table image

Trigger HITL if:

```text
Table structure confidence < 0.75
Column headers are ambiguous
Merged cells are detected
OCR confidence < 0.80
Extracted values are used in claims or KPIs
```

### Screenshot

Trigger HITL if:

```text
Screenshot contains application UI with important labels
Screenshot contains a document inside the document
Screenshot text density is high
OCR confidence < 0.75
System cannot classify the screenshot content
```

### Illustration or photo

Trigger HITL if:

```text
The visual covers more than 50% of the page or slide
The visual is referenced by surrounding text
The visual is likely not decorative
The model cannot determine semantic role
```

# 14. Recommended MVP

The MVP should not try to solve everything.

Recommended first implementation:

```text
1. Compute confidence at page / slide / chunk / visual level
2. Trigger HITL only for risky objects
3. Support visual HITL for diagrams, charts, and table images
4. Store human corrections
5. Reuse corrections to adjust thresholds later
```

MVP scoring dimensions:

```text
Text quality
Layout quality
Visual quality
Traceability
Hard HITL rules
```

Add later:

```text
Semantic consistency
Model agreement
Contradiction detection
Graph relationship confidence
Calibration from reviewer feedback
```

# 15. What not to do

Avoid these mistakes:

```text
Do not rely only on LLM confidence.
Do not score only the full document.
Do not send the full document to HITL when only one visual is risky.
Do not rely only on OCR for diagrams.
Do not treat captions as sufficient visual understanding.
Do not extract chart numbers without confidence and source validation.
Do not accept claims that cannot be traced back to a page, slide, chunk, or bounding box.
Do not use the same threshold for marketing, engineering, legal, and safety-critical content.
```

# 16. Final recommendation

Implement **Extraction Trust Score** as a core part of the semantic extraction platform.

The recommended production architecture is:

```text
Semantic + Structural Extraction Pipeline
+
Visual Semantic Extraction
+
Object-level Extraction Trust Score
+
HITL workflow for risky objects
+
Audit trail and feedback loop
```

The most important design choice is object-level review.

Do not send entire documents to human review by default.

Send only the risky elements:

```text
Pages
Slides
Visual zones
Tables
Charts
Diagrams
Claims
Requirements
KPIs
Decisions
Relationships
```

This keeps the system scalable while protecting quality.

# 17. Suggested internal naming

## Method name

**Extraction Trust Score — ETS**

## Short description

A composite confidence score that estimates whether extracted document knowledge can be trusted. It combines text quality, layout quality, visual extraction quality, semantic consistency, traceability, and model agreement. When the score is too low or risky patterns are detected, the system creates a Human-in-the-Loop validation task.

## Workflow name

**Trust-Gated HITL Extraction Workflow**

## Short description

A validation workflow that automatically routes low-confidence or high-risk extracted objects to a human reviewer. The reviewer validates, edits, or rejects the extraction, and the correction is stored for auditability and future calibration.
