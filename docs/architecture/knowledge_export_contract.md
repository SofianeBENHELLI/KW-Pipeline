# Knowledge Export Contract

> Status: accepted (2026-05-07).
>
> Closes [#23](https://github.com/SofianeBENHELLI/KW-Pipeline/issues/23).
>
> Scope-tightened per the 2026-05-04 backlog restructure: vector
> indexing has shipped under Phase 3, so this contract defines the
> *deterministic export package* downstream consumers (RAG indexers,
> data warehouses, customer handoff bundles) read — stable IDs,
> content-addressed checksums, and clear validation labelling. The
> *operator handoff button* that calls into this exporter is the
> separate ticket [#90](https://github.com/SofianeBENHELLI/KW-Pipeline/issues/90).

## Why a separate contract

The in-memory `SemanticDocument` and the Neo4j knowledge graph already
chunk validated documents — but neither shape is friendly to downstream
indexing on its own:

- `SemanticSection.id` is generated by the semantic extractor (UUID
  by default in tests, deterministic-but-opaque from the parser in
  practice). A consumer that re-imports a re-export cannot tell
  whether two ids point at the same content.
- The Neo4j projection is a queryable graph, not a portable artefact.
  Pulling RAG training data out of Cypher is operationally awkward
  and version-coupled.
- The semantic JSON carries free-text metadata that consumers may or
  may not need; without an explicit export schema, every consumer
  re-derives "which fields matter" and drifts.

The export package solves all three: a single self-describing JSON
artefact with content-addressed IDs and a top-level `package_sha256`
that lets consumers detect "nothing changed, skip re-index".

## Package layout

A `KnowledgeExportPackage` is one JSON object with three required
sections:

```jsonc
{
  "manifest": {
    "schema_version": "v0.1",
    "document_id": "doc-…",
    "document_version_id": "ver-…",
    "document_version_number": 3,
    "original_filename": "supplier-onboarding.pdf",
    "version_filename": "supplier-onboarding.pdf",
    "document_sha256": "…64-hex…",          // hash of the binary upload
    "content_type": "application/pdf",
    "semantic_schema_version": "v0.1",
    "validation_status": "validated",       // version-level FSM state
    "document_type": "policy",              // semantic profile
    "chunk_count": 12,
    "asset_count": 7,
    "package_sha256": "…64-hex…",           // hash of (chunks, assets)
    "exported_at": "2026-05-07T03:39:02.123456+00:00"
  },
  "chunks": [ /* see ExportedChunk */ ],
  "assets": [ /* see ExportedAsset */ ],
  "markdown": "…rendered markdown blob, optional…"
}
```

The `markdown` field is included verbatim from the `SemanticDocument`
because consumers that want a human-readable view (or want to feed
chunks alongside the rendered Markdown to a RAG pipeline) shouldn't
have to call a second endpoint.

### `ExportedChunk`

```jsonc
{
  "chunk_id": "chunk_a1b2c3d4e5f60718",       // deterministic; see below
  "section_id": "sect_…",                     // in-system SemanticSection.id
  "document_id": "doc-…",
  "document_version_id": "ver-…",
  "heading": "1.2 Eligibility",
  "text": "…",
  "char_count": 482,
  "content_sha256": "…64-hex…",               // sha256 of normalized text
  "source_reference_ids": ["src_p3_para0"],
  "validation_status": "validated"            // mirrors version-level status
}
```

- `chunk_id` is **content-addressed** at the (version, normalized-text)
  pair. Two re-exports of the same version produce identical
  `chunk_id` values; a re-extraction that rewrites the chunk text
  produces a new `chunk_id`. Consumers can use it as a stable upsert
  key.
- `section_id` is preserved so consumers can correlate back to
  in-system surfaces (knowledge graph nodes, review-workspace links).
- `validation_status` is mirrored from the version-level FSM, not the
  per-section state — chunks are atomic with their version under the
  current contract (a chunk cannot be validated independently of its
  version).

### `ExportedAsset`

```jsonc
{
  "asset_id": "asset_e7f8d9…",                // deterministic
  "asset_type": "policy_rule",
  "text": "…",
  "confidence": 0.92,
  "review_status": "validated",               // 4-state per-asset
  "source_reference_ids": ["src_p3_para0"],
  "content_sha256": "…64-hex…"
}
```

- `review_status` is the 4-state field on `SemanticAsset`
  (`needs_review`, `source_backed`, `validated`, `rejected`) — a
  finer-grained signal than the version-level `validation_status`.
- The exporter does **not** drop `needs_review` or `rejected` assets
  from the package; consumers decide which to index. Acceptance
  criterion 4 is satisfied by labelling, not omission.

## Deterministic ID generation

```text
chunk_id  = "chunk_"  + sha256(document_id + "/" + version_id + "/" + normalized_text)[:16]
asset_id  = "asset_"  + sha256(version_id + "/" + asset_type + "/" + normalized_text)[:16]
```

Where `normalized_text` is:

1. NFKC-normalize.
2. Strip leading/trailing whitespace.
3. Collapse all internal whitespace runs (spaces, tabs, newlines) to a
   single ASCII space.

This is stricter than equality on the raw `text` because re-extractions
sometimes change line-wrapping without changing meaning. Without
normalization a re-export would invalidate every consumer's cache on
every parser tweak.

The `[:16]` truncation gives 64 bits of entropy, which is well above
collision-tolerance for a single document's chunk count
(O(thousands)). The full sha256 lives on the chunk as
`content_sha256` for consumers that want a 256-bit handle.

The `(document_id, version_id)` salt prevents accidental ID collision
across documents that happen to share verbatim sections (e.g. boilerplate
preambles).

## Package-level checksum

```text
package_sha256 = sha256(canonical_json({
  "chunks": [chunks sorted by chunk_id],
  "assets": [assets sorted by asset_id],
}))
```

`canonical_json` is JSON with sorted keys, no whitespace, and UTF-8
encoding. Sorting at the top level means re-export order doesn't change
the hash; sorting within each chunk/asset is unnecessary because the
schema field order is fixed by Pydantic's serialization.

The hash is stored on the manifest *after* the chunks/assets are
serialized. Consumers compare `package_sha256` against their last-seen
value to short-circuit re-indexing.

## Validation status semantics

The export distinguishes three labels per acceptance criterion 4:

| Source | Field | Values |
|---|---|---|
| `DocumentVersion.status` (FSM) | `manifest.validation_status` | `needs_review`, `validated`, `rejected` |
| `SemanticDocument.validation_status` | inherited into chunks | same three values |
| `SemanticAsset.review_status` | `assets[*].review_status` | `needs_review`, `source_backed`, `validated`, `rejected` |

A consumer that wants "only trustworthy data" filters on
`manifest.validation_status == "validated"` *and*
`assets[*].review_status in {"validated", "source_backed"}`. The
exporter never silently drops content because the audit trail relies
on every reviewer decision being externally visible.

## Out of scope (filed elsewhere)

- An HTTP route surfacing the package (#90). The exporter today is a
  service callable from the admin code path; the customer-facing
  download button + auth gating ships under EPIC 5.
- A multi-document corpus export (per-document export only in v0.1).
- Vector embeddings in the package — Phase 3 ships those via the
  Neo4j HNSW index; embedding the floats inline would balloon the
  artefact and re-couple the export schema to the Voyage model
  identity.

## Versioning

`schema_version: "v0.1"` on the manifest and on each chunk/asset. A
breaking change bumps to `v0.2` and ships a one-step migrator next to
`load_semantic_document` so consumers reading older packages from
backups have a documented upgrade path.
